**tuimsg/messages.go additions:**

Add types for task graph data (tuimsg must NOT import internal/task or internal/git -- Manager maps types):

```go
// ResolvedTask is the TUI-side representation of a resolved task.
type ResolvedTask struct {
    ID          string
    Description string
    State       string   // queued, running, completed (DB state)
    Branch      string
    IsReady     bool
    IsBlocked   bool
    BlockedBy   []string // short IDs of blocking tasks
    Layer       int
}

// TaskGraphSnapshot holds the resolved task graph for TUI rendering.
type TaskGraphSnapshot struct {
    Tasks    []ResolvedTask
    HasCycle bool
}
```

Add `TaskGraph TaskGraphSnapshot` field to `StateSnapshot` struct.

**manager.go additions:**

1. Import `internal/task` and `internal/dependency` packages.
2. Add `taskStore *task.TaskStore` and `depStore *dependency.DepStore` fields to Manager struct. Initialize in NewManager (only if db != nil).
3. In `pollAllWorktrees()`, after worktree polling, add task resolution:
   - If db is nil, skip (return empty TaskGraph)
   - `tasks, _ := m.taskStore.List(ctx)` -- fetch all tasks
   - `deps, _ := m.depStore.ListAll(ctx)` -- fetch all deps
   - Build branchMerged map: for each task with a non-empty Branch and state != "completed", look up the task's repo path from the worktree states already collected, call `git.IsBranchMerged(ctx, repoPath, task.Branch, git.DefaultBranch(ctx, repoPath))`. Cache results to avoid redundant subprocess calls within one poll.
   - `resolved, err := task.Resolve(tasks, deps, branchMerged)`
   - Map resolved to `tuimsg.ResolvedTask` slice (Manager owns the mapping, same pattern as mapGitState)
   - Set `snapshot.TaskGraph = tuimsg.TaskGraphSnapshot{Tasks: mapped, HasCycle: err != nil}`
4. Add `mapResolvedTasks(resolved []task.ResolvedTask) []tuimsg.ResolvedTask` helper (maps internal types to tuimsg types, truncating BlockedBy IDs to 8 chars).

Performance guard: Only call IsBranchMerged for tasks where state is "queued" or "running" and branch is non-empty. Skip completed tasks.

**pkg/tui/panels/tasks.go:**

Create a TaskGraphPanel that renders the resolved task DAG as layered text:

```go
package panels

type TaskGraphPanel struct {
    width    int
    height   int
    focused  bool
    graph    tuimsg.TaskGraphSnapshot
    scrollY  int  // vertical scroll offset for overflow
}

func NewTaskGraphPanel() TaskGraphPanel
func (p *TaskGraphPanel) SetSize(width, height int)
func (p *TaskGraphPanel) SetFocused(focused bool)
func (p *TaskGraphPanel) SetGraph(g tuimsg.TaskGraphSnapshot)
func (p *TaskGraphPanel) ScrollUp()
func (p *TaskGraphPanel) ScrollDown()
func (p TaskGraphPanel) View() string
```

Rendering approach (in View()):
1. Group tasks by Layer (0, 1, 2, ...).
2. Render layers left-to-right. Each layer is a column of task boxes.
3. Each task box is a small bordered block (lipgloss):
   - Line 1: description (truncated to ~20 chars)
   - Line 2: state badge with color:
     - ready: green bold "[READY]"
     - blocked: red "[BLOCKED]"
     - running: yellow "[RUNNING]"
     - completed: dim "[DONE]"
   - Border color matches state (green/red/yellow/dim)
4. Between layers, draw arrow connectors: "--->" using box-drawing characters.
5. Vertical stacking within a layer for multiple tasks at same depth.
6. If graph.HasCycle, show a warning banner: "Warning: cycle detected in task dependencies"
7. If no tasks, show: "No tasks yet. Create one with: devctl tasks create <description>"
8. Handle overflow: if rendered height > panel height, use scrollY offset. Show scroll indicator.
9. Wrap in a border matching the DetailPanel style (lipgloss.RoundedBorder, borderColor based on focus).

Keep box widths fixed (e.g., 24 chars) so layers align cleanly. Total width per layer = box + arrow gap (~28 chars). If layers exceed panel width, show what fits with a "... +N more layers" indicator.
